## ⚠️ 重大注意：タイムゾーン仕様（必読）

**レポート開発を始める前に必ずこのセクションを読むこと。**
日付の取り扱いを誤ると、医師への提出レポートに「実際と異なる日付のデータ」が載る重大な問題が発生する。

### DBのタイムゾーン仕様

| フィールド | タイムゾーン | 説明 |
|-----------|------------|------|
| `meal_date`（mealsテーブル） | **JST** | Claudeがシステムプロンプトの「今日の日付」から設定。コードで明示的にJST取得している |
| `log_date`（weight_logs / steps_logsテーブル） | **JST** | 同上 |
| `recorded_at`（全テーブル） | **UTC** | SQLiteの`CURRENT_TIMESTAMP`がUTCで記録される |

### レポート開発時のルール

**✅ 日付の絞り込みには `meal_date` / `log_date` を使う**

```sql
-- ✅ 正しい：JSTの日付フィールドで絞り込む
WHERE meal_date BETWEEN '2026-03-15' AND '2026-03-21'
```

**❌ `recorded_at` で日付絞り込みをしてはいけない**

```sql
-- ❌ 禁止：recorded_at はUTCのため日付がJSTと最大9時間ずれる
-- 例：JSTで3/19 0:00〜8:59に記録されたデータは
--     recorded_at では 3/18 15:00〜23:59 になっている
WHERE recorded_at BETWEEN '2026-03-15' AND '2026-03-21'  -- 日付ずれが起きる
```

**`recorded_at` の用途**

- 同一日・同一食事区分内での並び順（`ORDER BY recorded_at`）にのみ使用する
- UIで記録時刻を表示する場合は `+9時間` してJST変換してから表示する

### 具体的な日付ずれの例

```
ユーザーがJST 2026-03-20 00:30 に夕食を記録した場合：
  meal_date  = 2026-03-20  （JST・正しい）
  recorded_at = 2026-03-19 15:30:00  （UTC・9時間前）

recorded_at で 2026-03-20 を検索すると → このレコードがヒットしない
recorded_at で 2026-03-19 を検索すると → このレコードが誤ってヒットする
```

## 1. 概要

### 目的
食事記録アプリのデータから、主治医（千葉大学医学部附属病院・糖尿病・代謝・内分泌科）への
定期提出用レポートをPDFとしてダウンロードする機能。
受診のたびに該当週を選択してPDFを生成・印刷して提出する。

### レポート単位
- **1週間単位**（日曜日〜土曜日）
- 例：2026年3月15日（日）〜2026年3月21日（土）

### 出力形式
- **PDF**（サーバー側で生成・ダウンロード）
- **用紙サイズ：A4横**
- **1週間分を1枚に収める**

## 3. 各セクションの詳細

### グラフエリア（レイアウト上部・横幅全体）

3つのグラフを横並びに配置：

| グラフ | 種類 | 内容 |
|--------|------|------|
| カロリー推移 | 折れ線グラフ | 1日合計kcal・目標ライン（1500kcal）も表示 |
| 体重推移 | 折れ線グラフ | 朝・夜を色分け（朝：青・夜：オレンジ） |
| 歩数推移 | 棒グラフ | 1日の合計歩数 |

X軸は全グラフ共通で日曜〜土曜の7日分。
グラフと表の日付が視覚的に対応するよう縦方向に揃える。

---

### 食事内容テーブル

| 行 | 内容 |
|----|------|
| 朝食 | 記録されたメニューのテキスト |
| 昼食 | 記録されたメニューのテキスト |
| 夕食 | 記録されたメニューのテキスト |
| 間食 | 記録されたメニューのテキスト（なければ空欄） |
| 夜食 | 記録されたメニューのテキスト（なければ空欄） |

同じ食事区分に複数レコードがある場合は改行して並べる。
記録がない日・食事区分は空欄。

---

### 栄養・バイタルテーブル

| 行 | 内容 | 単位 |
|----|------|------|
| Cal | 1日の合計カロリー | kcal |
| P/F/C | タンパク質/脂質/糖質 | g（例：P32/F18/C95） |
| 塩分 | 1日の合計塩分相当量 | g |
| 体重・朝 | 朝の体重 | kg |
| 体重・夜 | 夜の体重 | kg |
| 歩数 | 1日の合計歩数 | 歩 |

---

### メモ欄

- **手書き記入用の空欄**として十分な高さを確保
- PDF上では空白のまま出力
- 印刷後に手書きで追記する前提

---

### Claude補足欄

自動分析コメントを自動生成して記載。不要な場合は空欄。

**補足コメントの例：**
- 「今週はカロリーが目標を3日超過しました（平均1720kcal）」
- 「タンパク質が全体的に不足気味です（平均28g/日・目標60g）」
- 「体重が週を通じて減少傾向です（-1.2kg）」
- 「木曜日の記録が未入力です」
- 「塩分が高めの日が続いています（平均8.2g/日）」

複数該当する場合は箇条書きで列挙。
特記事項がない場合は空欄のまま。

## 5. PDF生成技術仕様

### 生成方法
サーバー側でPDFを生成してダウンロード（案B：見た目固定）。
ブラウザの印刷機能には依存しない。

### 推奨ライブラリ
**WeasyPrint**（Python）

```
理由：
- HTML/CSSからPDFを生成できる
- 日本語フォント対応
- A4横レイアウトをCSSで完全制御可能
- uvで管理可能
- ブラウザ環境に依存しないため見た目が完全に固定される
```

### フォント
日本語表示のため **Noto Sans CJK JP** を使用。
VPSにインストール済みの場合はそのまま利用。

### 依存パッケージ
```
weasyprint
```

## 6. データソース

レポートのデータはすべてメインアプリと共有のSQLite3 DBから取得。

> ⚠️ **日付絞り込みは必ず `meal_date` / `log_date`（JST）で行うこと。**
> `recorded_at`（UTC）を使うと最大9時間の日付ずれが発生する。

```sql
-- 食事内容の取得
-- WHERE句は meal_date（JST）で絞り込む ← ここ重要
SELECT meal_date, meal_type, description, calories, protein, fat, carbs, sodium
FROM meals
WHERE meal_date BETWEEN '2026-03-15' AND '2026-03-21'
ORDER BY meal_date, meal_type, recorded_at;  -- 並び順にrecorded_atは使ってよい

-- 体重の取得
-- WHERE句は log_date（JST）で絞り込む ← ここ重要
SELECT log_date, time_of_day, weight_kg
FROM weight_logs
WHERE log_date BETWEEN '2026-03-15' AND '2026-03-21'
ORDER BY log_date, time_of_day;

-- 歩数の取得
-- WHERE句は log_date（JST）で絞り込む ← ここ重要
SELECT log_date, steps
FROM steps_logs
WHERE log_date BETWEEN '2026-03-15' AND '2026-03-21'
ORDER BY log_date;
```

## 7. Claude補足コメントの生成

レポート生成時にClaude APIを呼び出して補足コメントを生成する。

### プロンプト方針
```
以下の1週間分の食事・体重・歩数データを分析し、
主治医への提出レポートに添付する補足コメントを生成してください。

【ルール】
- 医師が読むことを前提とした簡潔な日本語で記載
- 特記事項がない場合は空文字を返す
- 箇条書きで3項目以内
- カロリー目標：1500kcal/日
- 客観的なデータに基づいたコメントのみ（励ましや感想は不要）

【データ】
（週のサマリーデータをJSON形式で渡す）
```
