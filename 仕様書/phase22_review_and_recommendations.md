## C（コードレビュー）

### C-1: Step1 — イベント委譲の初期化時1回登録（評点: 9.5）

**修正内容の評価:** 正確かつ最小限の修正。`loadQuickPalette()` 内から `addEventListener` を削除し、ページ初期化部で1回だけ登録する変更は、イベント委譲パターンの正しい使い方。`innerHTML` による子要素の再生成はイベント委譲と相性が良く、リスナー再登録が不要になる設計。

**良い点:**
- 根本原因（リスナー蓄積）を正しく特定し、最小変更で解消
- 配置場所も適切（`passwordInput` のリスナー登録直後、`checkAuth()` 直前）
- コメント `// 1回だけ登録` で意図が明確

**指摘なし。**

### C-2: Step2 — ドラッグをハンドル限定に変更（評点: 9.5）

**修正内容の評価:** ウィジェット・サマリー両モーダルに対称的に同一パターンを適用。PC（HTML5 D&D）とスマホ（Touch Events）の両方をカバー。

**良い点:**
- `li.draggable = false` → `mousedown` で `true` → `dragend` で `false` のトグルパターンは、ハンドル限定D&Dの標準的な手法
- タッチイベントの条件反転（`!e.target.closest('.widget-toggle')` → `!e.target.closest('.widget-drag-handle')`）は明快。トグル以外を除外する消極的ガードから、ハンドルを要求する積極的ガードへの変更
- CSS の `cursor: grab` を `.widget-item` から `.widget-drag-handle` に移動し、視覚的アフォーダンスとインタラクション範囲を一致させている
- ウィジェット・サマリー両方で一貫した修正

**軽微な指摘:**

**C-2a（低優先度・改善提案）:** `mousedown` で `draggable = true` を設定した後、ユーザーがハンドルをクリックするだけ（ドラッグせず離す）場合、`dragend` は発火せず `draggable = true` が残る。次回はアイテム全体がドラッグ可能になる。実害は極めて限定的（モーダル内で連続操作する場面でのみ発生し得る）だが、完全を期すなら `mouseup` でも `draggable = false` に戻すとよい。

```javascript
// attachDragEvents / attachSummaryDragEvents 内、mousedown の直後に追加
handle.addEventListener('mouseup', function() { item.draggable = false; });
```

→ **対応任意。** 現状でもモーダルを閉じて再度開けばリセットされるため実用上問題なし。

## D（デザインレビュー）

### D-1: ハンドルアイコンのタッチターゲットサイズ（評点: 9.0）

現在の `☰` ハンドルは `font-size: 16px` で、周囲のパディングを含めてもタッチターゲットが小さめ。Apple HIG / Material Design のガイドラインでは最小 44x44px / 48x48dp を推奨。

**改善提案（低優先度）:**

```css
.widget-drag-handle {
  color: var(--text-secondary);
  font-size: 16px;
  flex-shrink: 0;
  cursor: grab;
  /* タッチターゲット拡大 */
  padding: 8px 4px;
  margin: -8px -4px;
}
```

→ **対応任意。** 現状の `padding: 10px 12px`（親要素 `.widget-item`）があるため、ハンドル付近のタップは概ね拾える。明確な操作困難の報告がなければ現状維持で可。

### D-2: ドラッグ中の視覚フィードバック（問題なし）

`cursor: grab` → ドラッグ中は `cursor: grabbing` に変わるのがベストプラクティスだが、ブラウザのデフォルトD&Dでは自動的にドラッグカーソルが表示されるため、現状で問題なし。タッチ操作では `.drag-over` クラスによるハイライトがフィードバックとして機能している。
