# トークン最適化・会話管理改善 開発計画

**作成日:** 2026-03-21
**対象バグ:** Claude API レート制限エラー（HTTP 429）
**バグ報告書:** `test/bug_report_claude_code.docx`

---

## 1. 問題の概要

### 根本原因

`claude_client.py` にて `conversation_history`（グローバルリスト）を毎リクエストで全件送信している。食事記録が進むにつれてトークン数が累積増加し、組織プランの上限（30,000入力トークン/分）を超過する。

**APIログで確認された実績：**
- セッション開始時の入力トークン：約 3,688
- レート制限到達時の入力トークン：約 32,983
- 1セッションで **約9倍**に増加

### 構造的な問題

| 問題点 | 詳細 |
|--------|------|
| 会話履歴の無制限増加 | `conversation_history` に上限なし |
| 全履歴の毎回送信 | 古い食事データがトークンを圧迫 |
| 履歴の非永続化 | サーバー再起動で消滅（UX問題） |
| エラーハンドリング未実装 | 429エラー時にユーザーへ生エラー表示 |

---

## 2. 要件・制約

### ユーザー要件

| 要件 | 内容 |
|------|------|
| ユーザー数 | 1名（複数端末からアクセス可）|
| 端末 | PC・スマートフォン・タブレット等の複数端末 |
| セッション管理 | サーバーサイドで管理（ブラウザ依存不可） |
| 会話継続範囲 | ログアウト・セッション切れまでは全履歴保持 |
| セッション切れ後 | 直近10件のみ引き継ぎ |
| 過去日付の参照 | ユーザーが過去日付を言及した際はDBから取得して対応 |
| 履歴圧縮 | 一定タイミングで圧縮しシステムプロンプトへ組み込む |
| **節約モード** | **設定画面でON/OFF切り替え可。トークン削減優先で過去文脈の精度が下がるモード** |
| プラン変更 | Anthropicプランのアップグレードはスコープ外 |

## 3. 設計方針

### 3-1. 会話コンテキスト構造（実装後）

```
API送信内容の構造：
┌─────────────────────────────────────┐
│ システムプロンプト（基本設定）         │ 約500-700トークン（固定）
│  └ 会話サマリー（圧縮済み履歴）       │ 約200-400トークン（動的追記）
├─────────────────────────────────────┤
│ 直近10件の会話（フル詳細）            │ 約2,000-5,000トークン
├─────────────────────────────────────┤
│ 新規メッセージ                       │ 約50-300トークン
└─────────────────────────────────────┘
目標合計: 3,000〜7,000トークン（現状の1/5以下）
```

### 3-2. 会話サマリーのハイブリッド圧縮方式

**圧縮トリガー条件：**
- 推定入力トークン数が **20,000を超えたとき**（APIリクエスト前に判定）

**圧縮対象：**
- 直近10件より古い全メッセージ

**圧縮手順：**
1. 古い履歴 + 既存サマリーを Claude（Haiku）に渡す
2. 「重要な決定事項・記録内容を箇条書きで要約」させる
3. サマリーをDBに保存
4. 古い履歴を `conversation_history` から削除
5. サマリーをシステムプロンプトの末尾に追記

**サマリー例：**
```
【会話サマリー（自動生成）】
- 朝食: ごはん200g、卵焼き（記録済み）
- 昼食の候補としてサラダチキンを検討中
- ユーザーの好みにより塩分表示を重視すると確認
```

### 3-3. セッション管理

**アクティブセッション中：**
- `conversation_history` はメモリ上でフル保持
- 一定トークン量を超えたら自動圧縮

**セッション切れ・ログアウト時：**
- 直近10件をDBに永続化（`conversation_messages` テーブル）
- 最新のサマリーをDBに保存（`conversation_summary` テーブル）

**次回セッション開始時：**
- DBから直近10件を復元
- 最新サマリーをシステムプロンプトに組み込む

### 3-4. 節約モード（Savings Mode）

設定画面でON/OFFを切り替えられる動作モード。APIコストを優先し、過去会話の精度を犠牲にする。

#### 標準モード vs 節約モード 比較

| 項目 | 標準モード（デフォルト） | 節約モード |
|------|------------------------|-----------|
| 直近履歴の保持件数 | 10件 | 3件 |
| 圧縮トリガー（推定トークン） | 20,000超 | 8,000超 |
| サマリー生成 | 詳細（箇条書き + 文脈情報） | 超簡潔（1行〜3行のみ） |
| 食事デフォルト一覧 | システムプロンプトに毎回含める | **含めない**（ツール検索時のみ取得） |
| 過去会話の文脈参照 | 直近10件まで自然に参照可 | 直近3件のみ。それ以前は不明 |
| 推定定常トークン数 | 3,000〜7,000 | **1,200〜2,500** |

#### トレードオフ（ユーザーへの説明文）

> **節約モードON時の制限：**
> - 「さっき入力した内容」が3件より前の場合、Claudeが文脈を把握できないことがあります
> - 食事のよく使うデフォルト情報はリクエスト時に都度検索されます（わずかな遅延が出ます）
> - 過去サマリーは短縮されるため、複数の食事変更を会話で連続して行う場合に精度が落ちることがあります

#### 食事デフォルトの扱い（節約モード固有）

通常モードでは `food_defaults` テーブルの内容をシステムプロンプトに毎回埋め込むが、節約モードでは省略し、ユーザーが食事名を入力したときのみ `search_food_nutrition` ツール経由で取得する。これだけで **約100〜300トークン/リクエストの削減**が見込める。

#### 設定の保存先

`app_settings` テーブルの既存の仕組みを利用：
```
key: "savings_mode"
value: "true" | "false"  （デフォルト: "false"）
```

#### UI（設定画面への追加）

`settings.html` にトグルスイッチを追加：
```
[ 節約モード ]  OFF ●━━━━ / ━━━━● ON
コストを優先します。直近3件の会話のみ保持されます。
```

---

### 3-5. 過去日付参照

既存の `get_daily_summary` ツールをそのまま活用。
システムプロンプトに以下の指示を追記：

> 「ユーザーが特定の日付に言及した場合（例：『3月15日の昼食』）、`get_daily_summary` ツールでDBから情報を取得してから回答すること。」

**節約モード時も同様に機能する**（過去日付はモード問わずDB参照）。

## 4. DBスキーマ追加

### 新規テーブル：`conversation_messages`

```sql
CREATE TABLE conversation_messages (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    role        TEXT NOT NULL,          -- 'user' | 'assistant'
    content_json TEXT NOT NULL,         -- JSON文字列（messagesブロック形式）
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### 新規テーブル：`conversation_summary`

```sql
CREATE TABLE conversation_summary (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    summary_text  TEXT NOT NULL,        -- 圧縮後のサマリー本文
    covered_up_to INTEGER,              -- どのmessage IDまで圧縮したか
    created_at    DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

## 5. 開発ステップ（実装順序）

### Step 1：エラーハンドリング ★即日対応
**目的：** UX改善・エラー時の体験向上
**対象ファイル：** `claude_client.py`

- [ ] 429エラー検知と指数バックオフリトライ（最大3回、1秒→2秒→4秒）
- [ ] リトライ中のSSEイベント通知（「混み合っています、しばらくお待ちください」）
- [ ] 最終失敗時の日本語エラーメッセージ表示
- [ ] その他APIエラー（500等）の日本語化

**実装イメージ：**
```python
async def call_with_retry(client, **kwargs, max_retries=3):
    for attempt in range(max_retries):
        try:
            return await client.messages.stream(**kwargs)
        except anthropic.RateLimitError:
            if attempt < max_retries - 1:
                wait = 2 ** attempt
                yield_status_event(f"混み合っています。{wait}秒後に再試行します…")
                await asyncio.sleep(wait)
            else:
                raise
```

## 7. 期待効果

| 指標 | 現状 | 標準モード（改善後） | 節約モード（改善後） |

| 定常入力トークン数 | 累積増加（~33,000到達） | 3,000〜7,000で安定 | **1,200〜2,500で安定** |

| 過去会話の参照精度 | （増加後に制限到達） | 直近10件は高精度 | 直近3件のみ高精度 |
