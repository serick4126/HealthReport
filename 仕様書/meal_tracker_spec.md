## 1. プロジェクト概要

### 目的
ユーザー（Serick）がチャット形式の自然言語・音声入力・画像送信で食事・体重・歩数を報告し、
Claude API が内容を解析してVPS上のSQLite3データベースに記録するWebアプリケーション。
主治医（千葉大学医学部附属病院・糖尿病・代謝・内分泌科）への定期レポート提出も目的の一つ。

### ユーザー情報
- 43歳男性 / 身長180cm / 体重約165kg / BMI56.9
- インスリン抵抗性を中心としたメタボリックシンドローム
- 主治医指示：1日摂取カロリー1500kcal・水分3L/日（糖分なし）・塩分管理
- 服薬：フェブキソスタット20mg（毎日）/ マンジャロ（チルゼパチド）処方予定
- 入力デバイス：iPad（Claudeアプリ・音声入力メイン）/ PC併用
- 夕食：コープデリ舞菜おかず利用（栄養成分表ラベルあり）

## 3. データベース設計

### テーブル1：meals（食事記録）

```sql
CREATE TABLE meals (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    recorded_at   DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    meal_date     DATE NOT NULL,
    meal_type     TEXT NOT NULL,
    description   TEXT NOT NULL,
    calories      INTEGER,
    protein       REAL,
    fat           REAL,
    carbs         REAL,
    sodium        REAL,
    notes         TEXT
);
```

**meal_type の値：**
- `breakfast`（朝食）
- `lunch`（昼食）
- `dinner`（夕食）
- `snack`（間食）
- `late_night`（夜食）

**notes の自動付記ルール：**
- 水分摂取量が報告された場合 → notesに記載
- Claude推定値の場合 → `[推定値] カロリー・PFCはClaudeによる推定`
- 料理写真から推定の場合 → `[写真より推定]`
- ラベル読み取りの場合 → `[ラベルより]`
- バーコードから取得の場合 → `[バーコードより]`

---

### テーブル2：weight_logs（体重記録）

```sql
CREATE TABLE weight_logs (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    recorded_at   DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    log_date      DATE NOT NULL,
    time_of_day   TEXT NOT NULL,
    weight_kg     REAL NOT NULL
);
```

**time_of_day の値：**
- `morning`（朝）
- `evening`（夜）

---

### テーブル3：steps_logs（歩数記録）

```sql
CREATE TABLE steps_logs (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    recorded_at   DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    log_date      DATE NOT NULL,
    steps         INTEGER NOT NULL
);
```

**備考：** 1日1レコード。同日に再報告された場合は上書き確認UIを表示。

---

### テーブル4：meal_images（食事画像）

```sql
CREATE TABLE meal_images (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    recorded_at   DATETIME DEFAULT CURRENT_TIMESTAMP,
    meal_id       INTEGER REFERENCES meals(id),
    image_data    BLOB NOT NULL,
    mime_type     TEXT NOT NULL,
    source_type   TEXT NOT NULL,
    notes         TEXT
);
```

**source_type の値：**
- `label`（栄養成分表ラベル）
- `photo`（料理写真）
- `barcode`（バーコード）

**画像処理仕様：**
- 長辺1200px以内にリサイズ
- JPEG品質80%で圧縮
- 圧縮後にSQLite BLOBへ保存
- 1回の送信で複数枚対応
- 記録確定後にmeals.idと紐付けて保存

---

### テーブル5：food_defaults（食品デフォルト設定）

```sql
CREATE TABLE food_defaults (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    created_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
    keyword       TEXT NOT NULL,
    description   TEXT NOT NULL,
    notes         TEXT
);
```

**データ例：**

| keyword | description |
|---------|-------------|
| 鶏胸肉 | 皮無し |
| 鶏むね肉 | 皮無し |
| オートミール | 味の素製のロールドオーツ |

**Claudeへの渡し方：**
全レコードをシステムプロンプトに毎回含める。
件数は数十件程度を想定しており、パフォーマンス上の問題はない。
表記ゆれ・ひらがな・カタカナ・漢字の違いもClaudeが柔軟に解釈する。

## 5. 選択肢UI仕様

チャット画面内でClaudeが選択肢を返す場合、ボタンUIとして描画する。
テキスト入力不要でタップ/クリックのみで選択できる。

### 選択肢が必要なケース

| ケース | 例 |
|--------|----|
| 商品のサイズ・量違い | ヤクルト65ml / 112ml |
| カロリーに影響する要素が不明 | 鶏むね肉（皮あり/なし） |
| 調理方法が不明 | 鶏むね肉（焼き/揚げ/茹で） |
| 修正対象レコードが複数 | 同日に同じ食事区分が複数ある |
| 同日同区分に既存レコードあり | 追加 / 上書き の確認 |

### 選択肢UIの構造

```
┌─────────────────────────────┐
│ ヤクルト1000（65ml）  47kcal │  ← 候補ボタン
└─────────────────────────────┘
┌─────────────────────────────┐
│ ヤクルト1000（112ml） 80kcal │  ← 候補ボタン
└─────────────────────────────┘
┌─────────────────────────────┐
│ 🔍 該当する商品がない        │  ← 常に最下部に表示
└─────────────────────────────┘
```

### 「該当する商品がない」選択後のフロー

```
┌──────────────────┐
│ 🔍 再検索する     │  → テキスト入力で再検索 → 再度選択肢表示
└──────────────────┘
┌──────────────────┐
│ ✏️ 手動で入力する │  → 商品名・カロリー・PFC・塩分を手動入力
└──────────────────┘
┌──────────────────┐
│ 🤖 Claudeに推定   │  → 推定値で記録・notesに[推定値]を付記
└──────────────────┘
```

## 6. カロリー・栄養情報の検索フロー

```
ユーザーが食品名を報告
    ↓
Step 1: カロリーSlism を検索
    ↓ 見つかった → 候補を選択肢表示（1件なら自動採用）
    ↓ 見つからない
Step 2: 文部科学省 食品成分データベースを検索
    ↓ 見つかった → 候補を選択肢表示（1件なら自動採用）
    ↓ 見つからない
Step 3: Web検索でオフィシャルサイトを探す
        （例：「商品名 カロリー 公式」）
    ↓ 見つかった → 候補を選択肢表示（1件なら自動採用）
    ↓ 見つからない
Step 4: 選択肢UIで「該当なし」フローへ
        （再検索 / 手動入力 / Claude推定）
```

**出典はユーザーに表示しない。最初に見つかったデータを採用。**
**複数候補がある場合（サイズ・量違いなど）は選択肢UIで表示。**

## 9. 会話フロー設計

### パターンA：食事報告（テキスト）
```
ユーザー：「今朝はオートミール50gとヤクルト1000飲みました」
→ meal_type: breakfast / 日付指定なし → 当日
→ カロリー・PFC・塩分を検索または推定
→ mealsテーブルに保存
→ 確認メッセージ返答

「✅ 朝食を記録しました
  メニュー：オートミール50g、ヤクルト1000
  220kcal / P:8g F:4g C:38g 塩分:0.2g」
```

### パターンB：体重報告
```
ユーザー：「今朝の体重は164.5キロでした」
→ time_of_day: morning / 当日
→ weight_logsに保存・前回比を表示

「✅ 朝の体重を記録しました
  164.5kg（前回朝比：-0.5kg）」
```

### パターンC：歩数報告
```
ユーザー：「今日は3200歩歩きました」
→ 当日のsteps_logsに保存

「✅ 歩数を記録しました：3,200歩」
```

### パターンD：複数まとめて報告
```
ユーザー：「昼はサンドイッチ1個とお茶500ml、夜の体重は165キロ」
→ meals（昼食・notesにお茶500ml）+ weight_logs（夜）をまとめて保存
→ まとめて確認メッセージ
```

### パターンE：修正（直近）
```
ユーザー：「さっきの朝食、オートミールじゃなくてロールドオーツ50gです」
→ 当日の最新breakfastレコードを更新

「✅ 朝食の記録を修正しました
  オートミール → ロールドオーツ50g」
```

### パターンF：修正・遡り入力（日付指定）
```
ユーザー：「3月18日の夕食は〜でした」
ユーザー：「3月18日の昼食を〜に修正して」
→ 日付を明示的に解釈（「3月18日」→ 2026-03-18）
→ 該当レコードを検索・保存/更新
```

### パターンG：画像送信
```
ユーザー：「今日の夕食です」＋ラベル写真（複数可）
→ 画像を圧縮・Claude APIで解析
→ 栄養成分表なら数値を直接読み取り記録
→ 料理写真なら推定値で確認UIを表示
→ バーコードならJANコードを読み取り商品検索フローへ
→ 記録確定後にmeal_imagesにも保存
```

### パターンH：食事提案（手動依頼のみ）
```
ユーザー：「昼ごはんの提案して」
ユーザー：「夕食何がいい？」
ユーザー：「朝食のメニュー提案して、コープデリで」
ユーザー：「間食に何かある？」

Claude処理：
  1. どの食事への提案かを発言から判断
  2. 当日の全記録（記録済みの全食事）を集計
  3. 残りカロリー・PFCバランス・塩分を計算
  4. 条件指定があればそれに従う
     条件指定なければ一般的な自炊メニューで提案
  5. 3候補程度を提案

例：「昼ごはんの提案して」の場合

「今日のここまでの摂取：320kcal（朝食のみ）
 残り：1180kcal

 昼食の提案：

 ① 鶏むね肉の塩焼き150g＋野菜サラダ＋味噌汁
    約380kcal / P:38g F:8g C:12g 塩分:1.8g
 ② 豆腐と野菜の炒め物＋ご飯小盛り
    約420kcal / P:18g F:12g C:55g 塩分:1.5g
 ③ 鮭の塩焼き＋ほうれん草のおひたし＋ご飯小盛り
    約450kcal / P:28g F:10g C:48g 塩分:2.0g」
```

## 10. システムプロンプト設計方針

```
あなたは食事記録アシスタントです。
ユーザーの発言から以下の情報を読み取り、
適切なツールを呼び出してデータベースに記録してください。

【記録対象】
- 食事（朝食/昼食/夕食/間食/夜食）
- 体重（朝/夜）
- 歩数（1日1回）

【ルール】
- 日付指定なし → 当日の日付を使用
- 日付指定あり → 明示された日付を使用（例：「3月18日」→ 2026-03-18）
- カロリー・PFC・塩分は検索フローで取得、見つからない場合は推定
- 推定値の場合はnotesに[推定値]を付記
- 水分摂取量が報告された場合 → mealsのnotesに記録
- 記録後は必ず確認メッセージを返す
- 曖昧な食事区分は文脈から推定、不明なら確認
- 修正依頼時は変更前・変更後の内容を確認メッセージに含める
- 食事と無関係な話題にも普通に応答する（記録操作は行わない）
- ユーザーの1日摂取カロリー目標は設定値を参照（デフォルト1500kcal）
- カロリーに影響する要素が不明な場合（皮あり/なし等）は選択肢UIを表示

【食品デフォルト設定の適用】
以下のキーワードに一致または類似する食品が報告された場合、
設定値を自動適用してください。
表記ゆれ・ひらがな・カタカナ・漢字の違いも柔軟に解釈してください。
（food_defaultsテーブルの全レコードをここに展開して渡す）
```

## 12. レポート機能（Phase 4・別途開発）

### 用途
主治医（千葉大学医学部附属病院）への定期提出用

### レポートに含める内容
- 期間指定（例：2026年3月1日〜3月31日）
- 日別カロリー合計
- 日別PFC合計
- 日別塩分合計
- 体重推移（朝・夜）
- 歩数推移
- グラフ画像

### レポートに含めないもの
- 食事画像（画像はアプリ内履歴のみ）
