## 1. 概要・方針

### 現状の問題点
チャット1ターンごとにシステムプロンプト全体（推定 1,500〜3,000 トークン）をフル料金で送信している。
10ターンの会話では、システムプロンプト分だけで 15,000〜30,000 トークンが発生する。

### キャッシングで解決できること
- **同一セッション内の2ターン目以降**：システムプロンプトのトークンコストが 90% 削減
- **1時間TTL（拡張）使用時**：セッション間（朝・昼・夜と時間を空けた利用）でもキャッシュヒット

### Batch APIについて
レポートコメント生成（Haiku月4回）への適用を検討したが、節約額が月0.004ドル（約0.6円）と効果が薄い。
**実装しない。**

## 2. Phase 1: 全体キャッシング

### 概要
システムプロンプト全体を1ブロックとして `cache_control` を付与する。
変更箇所が少なく、効果が大きい。**最優先で実装する。**

### キャッシュ構成

```
[system]
  └ Block 1: システムプロンプト全体  ←── cache_control: {"type": "ephemeral"}
```

### キャッシュヒット条件（5分TTL）
- 前回リクエストから5分以内に次のターンを送信すること
- システムプロンプトの内容が前回と完全一致すること

### キャッシュミスが発生する条件
| 条件 | 頻度 |
|------|------|
| 日付が変わった（`{today}` が変化） | 1日1回 |
| ユーザー設定変更（名前・目標カロリー等） | 稀 |
| food_defaults の追加・削除 | 稀 |
| 会話サマリーが更新された（圧縮発生時） | 数十ターンに1回 |
| 5分以上メッセージを送らなかった | セッション依存 |

日付変化は毎日1回発生するが、その後の全ターンはヒットする。
**1日10ターン送信した場合、9ターン分のシステムプロンプトが90%オフになる。**

### 変更ファイル

#### `claude_client.py`

**① `build_system_prompt()` の戻り値型を `str` → `list[dict]` に変更**

```python
# 変更前
def build_system_prompt(...) -> str:
    ...
    return f"""..."""

# 変更後
def build_system_prompt(...) -> list[dict]:
    ...
    prompt_text = f"""..."""  # 内容は変わらない
    cache_ttl = database.get_setting("cache_ttl") or "5min"
    cache_control = {"type": "ephemeral", "ttl": "1h"} if cache_ttl == "1hour" else {"type": "ephemeral"}
    return [{"type": "text", "text": prompt_text, "cache_control": cache_control}]
```

**② `stream_chat()` の `messages.stream()` 呼び出し：変更不要**
`system` パラメータは `str` でも `list[dict]` でも受け付けるため、そのまま動作する。

**③ `_compress_history()` の `messages.create()` ：変更不要**
圧縮用プロンプトはトークン数が少なく（数十〜数百トークン）、最小キャッシュトークン数に届かないためキャッシュ対象外。

#### `database.py`

初期設定に `cache_ttl` を追加（Phase 2 と共通）:

```python
("cache_ttl", "5min"),
```

### キャッシュ最小トークン数の注意

| モデル | 最小トークン数 |
|--------|--------------|
| Sonnet 4.6 | 1,024 |
| Haiku 4.5 | 2,048 |

現在のシステムプロンプトはfood_defaults込みで通常 1,200〜3,000 トークン。
**food_defaultsが0件の環境では Sonnet の閾値1,024を下回る可能性がある。**
その場合 `cache_control` を付与しても自動的にキャッシュされない（エラーにはならない）。

## 3. Phase 2: キャッシュTTL設定

### 概要
断続的な利用パターン（朝・昼・夜と時間を空けて使用）に対応するため、
設定画面でキャッシュTTLを「5分」「1時間（拡張β）」から選択できるようにする。

### TTL比較

| TTL | 書き込みコスト | 読み取りコスト | 有効期間 |
|-----|-------------|-------------|---------|
| 5分（デフォルト） | ×1.25 | ×0.1 | リクエストのたびにリセット |
| 1時間（拡張β） | ×2.0 | ×0.1 | リクエストのたびにリセット |

1時間TTLは書き込みコストが高いため、**同一セッション内でターン数が少ない場合は割高になりうる**。
目安: 1セッションで3ターン以上あれば1時間TTLの方が有利。

### 変更ファイル

#### `database.py`

```python
("cache_ttl", "5min"),  # "5min" or "1hour"
```

#### `main.py`

```python
EDITABLE_SETTINGS = {..., "cache_ttl"}
plain_keys = [..., "cache_ttl"]
```

#### `claude_client.py`

```python
def get_client() -> anthropic.AsyncAnthropic:
    api_key = os.getenv("ANTHROPIC_API_KEY") or database.get_setting("anthropic_api_key")
    if not api_key:
        raise ValueError("ANTHROPIC_API_KEY が設定されていません")
    cache_ttl = database.get_setting("cache_ttl") or "5min"
    extra_headers = {}
    if cache_ttl == "1hour":
        extra_headers["anthropic-beta"] = "extended-cache-ttl-2025-04-11"
    return anthropic.AsyncAnthropic(api_key=api_key, default_headers=extra_headers)
```

#### `static/settings.html`

「API・節約設定」セクションにTTL選択セレクタを追加:

```
[ キャッシュTTL ]  [5分（標準）▼]
                    ├ 5分（標準）
                    └ 1時間（拡張β）
```

1時間選択時の注意書き:
> ⚠️ ベータ機能。初回キャッシュの書き込みコストが2倍になりますが、
> 1時間以内に再利用できればトータルで節約になります。

## 4. Phase 3: 静的/動的分割キャッシング（上位最適化）

### 概要
システムプロンプトを **静的ブロック**（ルール・フォーマット定義）と
**動的ブロック**（日付・ユーザー情報）に分割する。

静的ブロックは日付をまたいでもキャッシュが効くため、
Phase 1・2 で効果が不十分な場合に実施する。

### キャッシュ構成

```
[system]
  ├ Block 1: 役割定義・記録ルール・フォーマット・検索フロー  ← cache_control あり
  └ Block 2: 今日の日付・ユーザー情報・food_defaults・サマリー ← cache_control なし
```

### 実装上の課題

現在の `build_system_prompt()` では「記録ルール」セクション内に `{today}` が2箇所含まれており、
このセクションを静的ブロックに分離するには以下のリファクタリングが必要:

```
変更前: 「日付指定なし → 今日（2026-03-22）の日付を使用」
変更後: 「日付指定なし → 今日の日付を使用（今日の日付は【今日の情報】セクションを参照）」
```

### Block 1 のトークン数確保

Sonnet の最小キャッシュトークン数は 1,024。
現在の静的ルール部分は推定 600〜800 トークンで**不足する可能性がある**。
足りない場合は、ツール説明文（TOOLS リスト）の一部をシステムプロンプトに組み込む、
または詳細なガイドライン文章を追加することで対応する。

### 実施判断基準

Phase 1・2 実装後に以下を確認:
- `cache_creation_input_tokens` が毎ターン発生している（= 毎回キャッシュミス）
- 日をまたいだ利用が多い

→ 上記に該当する場合のみ Phase 3 を実施。

## 6. コスト試算

### 前提
- システムプロンプト: 2,000 トークン
- 1セッション: 10ターン
- 1日: 2セッション（計20ターン）
- モデル: claude-sonnet-4-6（入力 $3.00/1M）

### 5分TTL（同一セッション内ヒット想定）

```
書き込み: 2,000 × 1.25 × 2セッション = 5,000トークン相当（×2 = 10,000）
読み取り: 2,000 × 0.1 × 18ターン = 3,600トークン相当
合計: 13,600トークン相当 / 日

キャッシュなし: 2,000 × 20ターン = 40,000トークン / 日
削減率: 約 66%
```

### 1時間TTL（セッション間もヒット想定）

```
書き込み: 2,000 × 2.0 × 1回 = 4,000トークン相当（1日1回の日付変化）
読み取り: 2,000 × 0.1 × 19ターン = 3,800トークン相当
合計: 7,800トークン相当 / 日

削減率: 約 80%
```

## 7. 注意事項・リスク

| 項目 | 内容 |
|------|------|
| **最小トークン数** | Sonnet: 1,024 / Haiku: 2,048。food_defaultsが0件だと閾値を下回る可能性あり。キャッシュされないがエラーにはならない |
| **1文字でもミス** | システムプロンプトに1文字でも差異があるとキャッシュ不一致 |
| **SSEストリーミング対応** | `messages.stream()` でも `cache_control` 付き system list は正常動作（SDK対応済み） |
| **Python 3.11対応** | `cache_control` 辞書はf-string外で定義するため問題なし |
| **キャッシュ効果の確認** | ストリーム終了後に `stream.get_final_message().usage.cache_read_input_tokens` で確認可能 |
| **1時間TTLのベータリスク** | `extended-cache-ttl-2025-04-11` はベータ機能。APIの仕様変更で無効化される可能性あり |

---

*Batch API は対費用効果が低いため本設計から除外。*
