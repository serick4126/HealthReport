# HealthReport プロジェクト概要

食事・体重・歩数を記録するシングルユーザー向け健康管理Webアプリ。
Claude APIをバックエンドで使用し、自然言語で記録できる。

## グローバル規則(最重要)

ユーザーからのあらゆる指示(プラン作成、開発、レビューなどほかすべて)が入力されたとき、もし要件や仕様に不明点がある場合は即座に後続処理を停止し、ユーザーへの質問事項を作成しすぐに問い合わせを実行する。
ユーザーからの返答を待たずほかの作業を進めることは禁止。なぜなら、ユーザーからの回答により影響範囲外として進めようとした内容に影響が及ぶ可能性があるため。
実装作業中および思考プロセス中も適用される。思考プロセス内で不明点や確認や判断が必要な事項が発生したら、思考を停止しユーザーへ問い合わせプロセスを実行する。
ユーザーからの回答が送信された後、後続処理を再開する。
この問い合わせプロセスの実行回数に制限はない。

## 重要な設計上の決定事項

- **シングルユーザー**：複数ユーザー対応なし。セッション管理はサーバーサイドのみ（ブラウザストレージ不使用）
- **会話履歴管理**：アクティブセッション中はフル保持。セッション切れ後は直近10件のみDBに保持
- **トークン圧縮**：推定トークン数が閾値を超えたら自動的にHaikuで圧縮してシステムプロンプトに注入
- **節約モード**：設定画面でON/OFF。保持3件・閾値8000トークン・food_defaults省略に切り替わる
- **エラーハンドリング**：429エラーは最大3回リトライ（指数バックオフ）後に日本語メッセージ表示

## コーディング規約（厳守）

以下のルールは Phase 8 以降のすべての開発で厳守すること。違反するコードはレビューで不合格となる。
コード例は [`仕様書/coding_standards_reference.md`](仕様書/coding_standards_reference.md) を参照。

### 1. テスト実装規約

- **HTML/ソースコード文字列マッチング禁止:** `assert 'xxx' in html` や行順序検証は不可。フォーマット変更で壊れ、動作でなく存在しか検証できない
- **動作を検証するテストを書く:** API呼び出し・関数直接呼び出し・モック使用が基本。HTMLのDOM構造テストはBeautifulSoupに限り許容
- **テスト作成チェックリスト:**
  1. ユニットテスト優先（関数の入力→出力を検証）
  2. 外部依存はモック（Claude API・Slism・ファイルI/Oは `unittest.mock.patch`）
  3. 境界値テスト必須（空入力・ゼロ・負数・超長文・特殊文字）
  4. エラーパスもテスト（例外・404・422）
  5. 一時DBを使用（`tempfile.NamedTemporaryFile(suffix=".db")` で隔離）
  6. テスト間の独立性（グローバル状態に依存しない。setup/teardownで初期化）

### 2. エラーハンドリング規約

- **サイレント障害禁止:** `except Exception: pass` は不可。汎用メッセージのみの表示も不可
- **Python:** 例外は必ず `logger` に記録してからフォールバック処理
- **JavaScript fetch():** HTTPエラー（401/500等）では例外をスローしない。ステータス判定は `res.ok` / `res.status` で行い、ネットワーク障害は `catch(e)` で処理する2段階構成
- **main.py:** 内部エラー詳細をクライアントに露出禁止。`logger.error()` に記録し、汎用メッセージを返す

### 3. フロントエンド品質規約

- **Observer/EventListenerのクリーンアップ必須:** `IntersectionObserver`・`MutationObserver` は変数に保持し、再作成時に `disconnect()`。`beforeunload` 時にも解除
- **グローバル状態はオブジェクトに集約:** バラバラの `let` 変数禁止。`const pageState = {...}` に集約
- **動的生成時のonclick属性インラインJS禁止（XSSリスク）:** ユーザーデータは `data-*` 属性経由で渡す

### 4. Python アーキテクチャ規約

- **関数は100行以下:** 超過する場合は責務を分割。`claude_client.py` の `stream_chat()` は段階的に分割すること
- **ディスパッチテーブルを使う:** if-elif チェーンが5分岐以上になる場合はディクショナリによるディスパッチに置き換える
- **関数の置換・廃止時は全呼び出し元を列挙してから着手する:** 旧関数を `grep` し呼び出し元を一覧化。データが入る全経路（チャット、履歴ページ、外部API等）で新コードパスが使われることを確認する。レビュー時も旧関数名で `grep` し未修正の残存呼び出しがないことを検証する（Phase 12.1 教訓: 画像保存の2経路のうち主経路が修正漏れ）

### 5. 入力バリデーション規約

APIエンドポイントで外部入力を受け取る場合、以下を検証する:

- **文字列:** 最大長チェック（description: 500文字、notes: 1000文字、message: 10000文字）
- **数値:** 妥当な範囲チェック（calories: 0-9999、weight: 20-300、steps: 0-999999）
- **日付:** `YYYY-MM-DD` 形式の検証（正規表現 + datetime.strptime）
- **Base64画像:** デコード後サイズの上限チェック（10MB以下）

## 開発プラン・レビュー ワークフロー規約

### ファイル管理

- 保存先: `仕様書/` ディレクトリ（マークダウン形式）
- ファイル名規則: `phase{N}_development_plan.md` / `phase{N}_review_and_recommendations.md`
- 小数点フェーズの場合: `phase7_1_development_plan.md`

### 開発プラン（SONNETモデル担当）

- フェーズごとに1ファイルにまとめる
- レビューファイルが存在する場合はその内容に従う
- **ステップ構成順序:**
  1. 最重要の前提開発（共通コンポーネント等）を最初に配置
  2. 以降は重要度順（新規開発はプラン作成時判断、レビュー起因はレビュー内の重要度に従う）
- **各ステップ完了時に実施結果を追記:** 修正日・修正ファイル・修正内容・レビュー番号（あれば）・テスト結果
- **全ステップ完了時にサマリーを末尾に追記:** 開発内容・レビュアーへの報告事項・質問事項・レビュー指示と異なる方針を取った場合の理由等

### プラン作成・開発時の必須チェック事項

**プラン作成時:**
- 各ステップを記述するたびに、メモリおよびCLAUDE.mdの制約（コーディング規約・ワークフロー規約）と一致しているか確認すること
- プランファイルの**タイトル直下（概要より前）に `## 目次` セクションを配置する**。ファイルを開いた瞬間に全ステップが一覧できる位置でなければならない。目次の各行はコミットメッセージのテンプレートとして使用するため以下の形式に従うこと:

```
## 目次
| Step | ファイル | 内容 |
|------|---------|------|
| Phase{N}-Step1 | {ファイル名} | {変更内容} |
| Phase{N}-Step2 | {ファイル名} | {変更内容} |
...
```

  - `{ファイル名}` は複数ファイル記載可・要約可
  - 概要・設計思想等のセクションより**必ず前**に置くこと（複雑なチェック仕様はチェック漏れを招くため）

**開発時（各ステップ完了時）:**
- 目次のその行の内容をコミットメッセージとして `git commit` すること（形式: `Phase{N}-Step{M}\t{ファイル名}\t{変更内容}`）
- **pushは禁止**（commitのみ許可）
- コミット前に再度、メモリおよびCLAUDE.mdの制約と実装が一致しているか確認すること

### レビュー（OPUSモデル担当）

3つの視点で実施し、プリフィックス+番号で管理する:

| プリフィックス | 視点 | 内容 |
|-------------|------|------|
| **C** | コードレビュー | 厳格なプロフェッショナルとしてのコードレビュー |
| **D** | デザインレビュー | プロフェッショナルWEBデザイナーとしてのUI/UXレビュー |
| **U** | ユーザーレビュー | 一般ユーザー視点のUXレビュー＋不足機能提案 |

- **番号例:** C-1, D-1, U-7
- **評点:** 0〜10点（10点満点）。評価コメントと評点を記載
- **再評価:** 前回レビュー結果による修正プランの場合、修正後の再評価を行う
- **修正指示必須:** 開発担当（SONNET）に対して修正コードや修正方法を具体的に指示する
- **注意:** 開発モデルとレビューモデルが異なるため、指示内容の漏れに注意深く対応すること
- **git log注意:** `.gitignore` に含まれるファイル（テストプログラム等）の変更はgit logで取得不可
